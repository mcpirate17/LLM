## Current blocker — leaderboard upsert duplicate-fp guard

Direct end-to-end smoke test produces this error AFTER the new
`program_results` row writes successfully:

```
fingerprint a7e9003e76dbd513 is already on the leaderboard at
entry_id=210b4010-7e2 (result_id=4ea7a15d-b60); attempted insert
for result_id=0173a72a-923. Call promote_to_tier() on the existing
entry, or pass allow_fingerprint_duplicate=True if the duplicate is
intentional.
```

The replay's `_record_leaderboard_and_best` → `_upsert_screening_entry`
→ `nb.upsert_leaderboard` is trying to INSERT a new leaderboard row
keyed on the new `result_id`. The `idx_leaderboard_fp` UNIQUE partial
index rejects because a leaderboard row already exists for that fp
under a different `result_id`.

**Subtle:** the fp in the error (`a7e9003e76dbd513`) does NOT match the
source fp `c9c7075e741a8790`. That suggests `compile_model([graph] *
n_layers, ...)` produces a different fingerprint than the unstacked
graph, OR canonicalization is normalizing the graph during replay.
Worth verifying — the leaderboard parent should stay tied to the
source fp, not the layered model's fp.

## Plan A — minimal fix (preferred)

Make the replay's leaderboard upsert path:
1. Detect "this is a score-stability rerun" (via the same source_context check that gates `independent_sample`).
2. Look up the **existing** leaderboard entry by `graph_fingerprint`.
3. Re-upsert that existing entry's `result_id` so its aggregator picks up the new sample.

Concretely: in `_record_leaderboard_and_best` (or a new helper), if
`program_metrics["intentional_independent_sample"]` is true, replace
`rid` with `nb.get_leaderboard_entry_by_fingerprint(fp).result_id`
before calling the upsert. The existing entry's metrics will
re-aggregate via the per-fp aggregator I built earlier (it now reads
all `program_results` for the fp, computes mean/CV).

## Plan B — even simpler

For score-stability reruns, **skip the leaderboard upsert entirely** in
the replay path. The next time the leaderboard's existing entry gets
upserted (continuous tick, manual rescore, any other event), it'll
re-aggregate naturally because the new program_results row is in the
DB.

Cost: a delay between writing the new sample and seeing it reflected
in the leaderboard composite. We can mitigate by triggering a manual
rescore for the source fp at the end of the replay (one-line call).

## Plan C — fix the fingerprint mismatch first

If `compile_model([graph]*n_layers)` is producing a different
fingerprint than the source graph, the upsert is genuinely trying to
add a different architecture's row. That's a separate bug — the
provenance should be tied to the source fp, not the layered model.

We'd need to verify by logging both fingerprints during replay. If
they differ, decide whether to (a) record the source fp on the new
row and let the leaderboard reuse the existing entry, or (b) accept
that the fingerprint legitimately changed and create a fresh entry.

For score-stability the answer is clearly (a) — same architecture,
just another sample.
