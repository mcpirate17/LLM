### 2026-04-24 — Codex training performance audit

Compared against the other Codex notes in this file, repo-root `.current_work.md`
entries from 2026-04-19 01:00Z and 01:30Z, and `research/eval/WORKING.md`.
Those already cover the broad runtime/scientist passes, prior training native
consolidation, mandatory native CE/grad clipping, native RigL mask update, Muon
NS temporary reduction, and the existing fast data pipeline work. Not duplicated
here.

Scope actually audited this pass: `research/training/`.

Profiling evidence gathered:
- repeated small/medium/large wall-clock benchmarks
- `cProfile` cumulative and self-time
- `line_profiler` on the top Python paths
- `torch.profiler` CPU op and memory table
- `tracemalloc` allocation sampling
- startup separated from steady-state native-extension paths

Findings:
- `research/training/data_pipeline.py` was not the remaining bottleneck.
  Existing native JSONL/text tokenization, zero-copy `.npy`, and native batch
  gather measured at <= 1.75 ms uncached `.npy` load for 2M tokens and about
  0.07 ms steady-state CPU batch sampling.
- `research/training/curriculum.py::CurriculumStrategy.get_seq_len` was the
  clearest Python hotspot. In the representative profile it consumed 0.198s of
  0.315s with 500k calls, scaling linearly from 246 ms / 100k calls to 4.80s /
  2M calls under the full profiling harness.
- Synthesized losses remain the next serious compute target. Large
  all-components loss measured 25.6 ms; spectral-only measured 31.7 ms.
  `torch.profiler` showed `aten::topk` as the dominant self-CPU op, with
  `exp`, `log_softmax`, `mul`, and `pow` allocating tens of MB per medium
  representative run.
- Muon remains expensive at large matrix sizes: two 768x768 matrices measured
  112.6 ms per step. That is mostly real matmul work, but if Muon is frequent
  it should move toward a fused C++/CUDA optimizer path rather than more Python
  wrapping.
- RigL topology update is already partly native and lower ROI unless update
  cadence is high: one 768x768 matrix measured 8.74 ms.
- Training program synthesis is Python object churn but not the top runtime
  target: 138.6 ms for 4000 programs, about 4.9 MB Python allocations.

Changes landed:
- Added cached schedule metadata in `research/training/curriculum.py` for the
  scalar path.
- Added native C++ schedule precompute:
  `research/training/_curriculum_native.py` and
  `research/training/_curriculum_native.cpp`.
- Wired native precomputed schedules into
  `research/scientist/runner/execution_training_program.py` via
  `research/scientist/runner/_curriculum_schedule.py`, with fallback for legacy
  test/fake curricula that only expose `get_seq_len`.
- Added parity coverage in
  `research/tests/test_loss_optimizer_synthesis.py::test_curriculum_native_schedule_matches_scalar_path`.
- Added runner helper coverage in
  `research/tests/test_execution_training_native_boundary.py`.
- Fixed CPU native `rank_weighted_ce` so it no longer returns a detached scalar.
  The rank scan still runs in C++, but the final loss composes native rank
  weights with `log_probs.gather(...)` so synthesized losses with
  `rank_weighted_ce` as the primary component backpropagate correctly.
- Moved `contrastive_push` orchestration into `research/training/_loss_native.cpp`
  using ATen operations, preserving autograd while reducing Python dispatch and
  small allocations.
- Replaced `gradient_penalty`'s full-matrix `pow(2).mean()` temporary with a
  norm-squared reduction in `research/training/_loss_components.py`, preserving
  gradients and avoiding one large elementwise allocation.
- Simplified label-smoothed CE and KL-uniform reductions to use a single global
  `log_probs.mean()` instead of row-wise mean followed by another mean. Direct
  component timings were roughly flat to mildly faster; kept because it removes
  an intermediate row tensor without changing gradients.
- Tried a native custom-autograd entropy regularizer. Reverted it because the
  forward was roughly flat and backward was slower, worsening the full
  all-components benchmark.
- Tried a synthesized-loss dispatch cleanup, measured no real win, and reverted
  it before finishing.

Measured gains for curriculum scheduling:
- scalar path with profiling harness: 100k calls 246.17 ms -> 161.11 ms
  (34.6% faster); 500k calls 1215.12 ms -> 813.57 ms (33.0% faster); 2M calls
  4798.47 ms -> 3249.58 ms (32.3% faster).
- native `seq_len_tensor` path: 100k calls equivalent -> 0.66 ms; 500k -> 3.15
  ms; 2M -> 12.50 ms. This is the path to wire into loops that can consume a
  precomputed schedule.
- integrated precompute + per-step list indexing path: 100k schedule updates
  246.17 ms -> 131.75 ms (46.5% faster); 500k 1215.12 ms -> 667.78 ms
  (45.0% faster); 2M 4798.47 ms -> 2004.03 ms (58.2% faster). This is lower
  than pure native generation because the training loop still has one Python
  iteration per step.
- rank-weighted CE now preserves autograd. Forward vs Python reference:
  small 0.268 ms -> 0.188 ms (30.1% faster), medium 0.932 ms -> 0.540 ms
  (42.0% faster), large 11.38 ms -> 3.13 ms (72.4% faster). Medium backward
  was comparable: Python reference 2.03 ms, native-weight path 1.95 ms.
- contrastive push direct gains were modest because `topk` dominates:
  medium 1.015 ms -> 0.968 ms (4.6% faster), large 4.91 ms -> 4.88 ms
  (0.7% faster).
- full synthesized all-components loss improved on the large benchmark from
  the audit baseline 25.62 ms -> 19.56 ms (23.7% faster), while fixing the
  rank-weighted primary-gradient failure. The intermediate after rank/contrastive
  pass was 21.78 ms.
- The final profiler snapshot after these loss fixes is dominated by real ATen
  tensor kernels (`topk`, `log_softmax`, `exp`, elementwise reductions). Further
  meaningful loss gains require a fused C++/CUDA custom-autograd loss bundle,
  not more Python wrapper cleanup.

### 2026-04-24 — Codex synthesis performance audit addendum

Compared against the `Codex — research runtime performance audit pass 1` entry
above before adding this. That entry already records the broad
`graph_generation` wall-clock improvement, motif compatibility caching in
`_template_helpers.py`, and cached per-op parameter estimates in
`graph_ir_builder.py`, so those are not re-claimed here.

Additional synthesis-specific findings not duplicated above:
- `research/synthesis/_selection_utils.py::context_pair_allowed()` was importing
  context rules inside a high-frequency helper. In the targeted 250-graph
  profile this helper was called about 135k times before the hoist. Moving the
  `CONTEXT_RULES` import to module load removed import-system churn from motif
  compatibility checks.
- `research/synthesis/grammar_support.py::blend_template_weights_with_db()`
  was recomputing the same template/DB blend per generated graph. It showed up
  as `0.082s` cumulative in the baseline 250-graph profile. Caching the pure
  blend result cut that path to about `0.032s` cumulative.
- `research/synthesis/graph.py::{depth,n_params_estimate,has_gradient_path}` was
  building an extra reachable IR even though the existing analysis IR already
  carries output-reachable structure, depth, params, and gradient-path summary.
  Reusing `_analysis_ir()` reduced IR build calls in the 250-graph profile from
  about `2672` to `1596`.
- `research/synthesis/native_param_formula.py` reset ctypes arg/restype on every
  formula call and did not cache deterministic formula results. The baseline
  line profile saw about `15.9k` formula calls in the 200-graph run. Binding the
  signature once in `native_analysis_bindings.py` and adding an LRU cache made
  this path disappear as a meaningful line-profile cost.
- Startup is separate from steady-state: importing
  `research.synthesis.grammar`, `research.synthesis.graph`, and
  `research.synthesis.templates` measured about `1.10s` wall and `693 MB` max
  RSS in this environment. The landed changes improve repeated generation and
  validation, not import-time cost.

Targeted evidence:
- Baseline 250-graph `cProfile`: `1.054s`, `3.75M` calls.
- After targeted synthesis changes: `0.884s`, `2.70M` calls.
